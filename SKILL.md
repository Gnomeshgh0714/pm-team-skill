# 多Agent产品经理团队

## 概述

将单次产品迭代拆解为标准阶段，由5个专业角色协作完成。每个角色有独立的职责边界、输出规范和红线禁令。通过评审攻防机制确保方案质量，通过决策日志确保全程可追溯。

支持两种运行模式：**完整模式**（6阶段，适合从零做产品）和**快速模式**（4阶段，适合中小需求）。

## 何时使用

- 从零开始做一款产品（完整模式）
- 对已有产品进行一轮结构化迭代（完整模式）
- 单个功能模块的需求设计（快速模式）
- 需要多角色协作产出高质量产品方案（任意模式）
- 需要可追溯的决策记录和复盘沉淀（完整模式）

## 不适用场景

- 单一文档撰写（直接写即可）
- 纯技术开发任务（不涉及产品决策）
- 5分钟内能搞定的小修改

## 运行模式

### 完整模式（6阶段，适合从零做产品）

```
① 用研输入 → ② 产品设计 → ③ 评审攻防 → ④ 总监终审 → ⑤ 落地执行 → ⑥ 复盘优化
```

| 阶段 | 执行者 | 交付物 | 可并行 |
|------|--------|--------|--------|
| ① 用研输入 | 用户研究专员（子Agent） | 用研报告 | 可与增长方案并行 |
| ② 产品设计 | 产品设计专员（子Agent） | PRD+流程图+数据结构 | 串行（依赖①） |
| ③ 评审攻防 | 产品评审专员（子Agent） | 评审意见 | 串行（依赖②） |
| ④ 总监终审 | 产品总监（主Agent） | 终审决策+修改指令 | 串行（依赖③） |
| ④b 修改+复审 | 设计专员→评审专员 | 修改后PRD+复审意见 | 串行（依赖④） |
| ⑤ 落地执行 | 产品设计专员（子Agent） | 技术实施方案 | **可与⑥并行** |
| ⑥ 复盘优化 | 产品总监（主Agent） | 复盘报告 | **可与⑤并行** |

### 快速模式（4阶段，适合中小需求）

```
① 需求梳理 → ② PRD设计 → ③ 评审攻防 → ④ 总监终审定稿
```

| 阶段 | 执行者 | 交付物 |
|------|--------|--------|
| ① 需求梳理 | 主Agent直接执行 | 需求清单（用户画像简版+痛点+优先级） |
| ② PRD设计 | 产品设计专员（子Agent） | PRD文档 |
| ③ 评审攻防 | 产品评审专员（子Agent） | 评审意见 |
| ④ 总监终审 | 主Agent直接执行 | 终审决策（通过/打回） |

**快速模式特点**：
- 跳过技术实施方案和复盘
- 需求梳理由主Agent直接做，不调度子Agent
- 评审标准不降（仍≥3条问题）
- 适合P0功能已明确、不需要深度用研的场景

### 模式选择规则

| 条件 | 推荐模式 |
|------|---------|
| 新产品/新方向，需用研 | 完整模式 |
| P0功能已明确，只需PRD+评审 | 快速模式 |
| 需要技术方案+复盘 | 完整模式 |
| 时间紧迫，快速验证 | 快速模式 |

## 角色架构

### 产品总监（由主Agent担任）
- 全局把控、最终决策、团队管理
- 对评审意见逐项裁决，输出修改指令
- 每轮迭代后组织复盘，提炼可沉淀技能
- **质量把关**：对子Agent产出做质量验收，不达标则打回重做

### 4个专业角色（通过子Agent调度执行）

| 角色 | 职责 | 输出目录 | 命名规范 |
|------|------|---------|---------|
| 用户研究专员 | 用户画像、痛点分析、竞品分析、需求排序 | /research/ | 日期_主题_调研报告.md |
| 产品设计专员 | PRD撰写、数据结构设计、交互流程图、技术方案 | /product/prd/ | 版本号_产品名称_PRD.md |
| 产品评审专员 | 反向评审、风险识别、漏洞挖掘（只提问题不给方案） | /review/ | 版本号_PRD评审意见.md |
| 增长运营专员 | 指标体系、冷启动方案、版本运营、数据分析 | /growth/ | 日期_主题_方案.md |

## 子Agent上下文传递规范

子Agent无法读取主Agent的对话上下文，因此每次调度子Agent时必须传递**结构化交接信息**。

### 交接信息格式

主Agent调度子Agent时，在context字段中包含以下信息：

```
## 项目上下文
- 项目根目录：[路径]
- 产品方向：[一句话描述]
- P0功能：[列表]
- P2禁令：[列表]
- 当前阶段：[阶段X]
- 前序产出物：[文件路径列表]

## 本轮任务
- 你的角色：[角色名]
- 任务目标：[具体目标]
- 交付物路径：[输出文件路径]
- 交付标准：[具体标准]

## 下游重点
- 你的产出将被[下游角色]使用
- 下游特别关注：[需要重点确保的方面]

## 已知约束
- [如有特殊约束则列出]
```

### 文件传递原则
- 所有正式产出必须写入文件，不依赖对话传递
- 子Agent先读取前序产出文件，再执行任务
- 交接信息中只放摘要和路径，不放完整内容

## 质量门槛与失败处理

### 质量门槛（每阶段交付前检查）

| 阶段 | 门槛标准 | 不达标处理 |
|------|---------|-----------|
| 用研输入 | 痛点有场景+竞品≥3个+优先级有评分模型 | 打回重做，最多2次 |
| 产品设计 | 5项自检全通过+每个功能7要素齐全 | 打回重做，最多2次 |
| 评审攻防 | ≥3条实质性问题+结论明确 | 评审不合格，重新评审 |
| 技术方案 | 任务拆解含依赖关系+时间节点合理 | 打回补充，最多1次 |
| 复盘报告 | ≥1优化点+≥1可沉淀技能 | 打回补充，最多1次 |

### 失败处理流程

```
子Agent产出 → 总监验收
  ├─ 通过 → 进入下一阶段
  ├─ 不达标（第1次）→ 返回修改意见，重新调度
  ├─ 不达标（第2次）→ 返回修改意见，重新调度
  └─ 不达标（第3次）→ 总监直接修改，记录"人工介入"标记
```

> 子Agent重新调度时，在context中附加上次产出的问题说明，避免重复犯错。

## 阶段详细说明

### 阶段1：用研输入（完整模式）
- **执行方式**：调度用户研究专员子Agent
- **输入**：产品方向、P0/P1/P2功能边界
- **交付物**：用研报告（用户画像 + 痛点分析 + 竞品分析 + 需求优先级排序）
- **交付标准**：痛点有具体场景支撑，竞品至少3个，优先级排序有评分模型
- **并行说明**：可与增长运营方案并行（如果增长方案不依赖用研结果）

### 阶段1：需求梳理（快速模式）
- **执行方式**：主Agent直接执行
- **交付物**：需求清单（用户画像简版 + 核心痛点 + 功能优先级）
- **特点**：不调度子Agent，10分钟内完成

### 阶段2：产品设计（含提交前自检）
- **执行方式**：调度产品设计专员子Agent
- **输入**：用研报告（完整模式）或需求清单（快速模式）
- **交付物**：PRD文档 + 交互流程图 + 数据结构设计
- **交付标准**：每个功能含功能描述/用户故事/功能拆解/输入输出/交互逻辑/边界条件/验收标准
- **提交前自检清单**（5项最低）：
  1. 数据模型是否覆盖所有引用的实体类型（无悬空外键）
  2. 状态机是否所有状态有出口、无死锁/不可达路径
  3. 跨实体引用的外键是否指向已定义实体
  4. 边界声明与实际功能实现是否一致
  5. 并发场景是否定义数据一致性策略

### 阶段3：评审攻防
- **执行方式**：调度产品评审专员子Agent
- **输入**：用研报告 + PRD文档
- **交付物**：评审意见（问题编号 + 级别 + 类别 + PRD位置 + 问题描述）
- **红线规则**：
  - 至少3条实质性问题，否则评审不合格
  - 只提问题不给方案（改进方向由总监裁决）
  - 禁止无原则通过
  - 每条问题必须对应PRD具体位置

### 阶段4：总监终审
- **执行方式**：主Agent直接执行
- **输入**：评审意见
- **交付物**：终审决策（逐项裁决 + 修改指令分类 + 决策日志记录）
- **裁决方式**：对每条问题采纳/部分采纳/不采纳，附理由
- **修改指令分类**：
  - 必须修改（阻断性）：不修改不能进入下一阶段
  - 建议修改（非阻断）：强烈建议但非阻断
- **修改后流程**：设计专员修改 → 评审专员复审 → 总监二次终审

### 阶段5：落地执行（仅完整模式）
- **执行方式**：调度设计专员子Agent
- **输入**：定稿PRD
- **交付物**：技术实施方案（技术选型 + 架构设计 + 任务拆解 + 时间节点）
- **并行说明**：可与阶段6（复盘）并行执行

### 阶段6：复盘优化（仅完整模式）
- **执行方式**：主Agent直接执行
- **交付物**：复盘报告
- **必须包含**：至少1个可优化点 + 1个可沉淀技能
- **并行说明**：可与阶段5（技术方案）并行执行

## 关键机制

### 评审攻防机制
评审专员必须提出实质性反对意见。这一机制确保方案在进入开发前经过严格检验。

### 决策留痕
所有跨角色裁决、方案取舍、边界决策写入 /team-rules/决策日志.md，包含决策背景、内容、理由、生效时间。

### 技能沉淀
每轮迭代复盘后，将可复用的工作方法沉淀为Skill文件，存入 /team-rules/skills/。

## 能力升级路径

| 等级 | 标准 | 验收方式 |
|------|------|---------|
| L1 | 按流程走完单个需求，不跳步 | 完整走完"需求→PRD→评审→定稿" |
| L2 | 自主发现问题、优化质量 | 评审攻防有效，严重问题占比≥20% |
| L3 | 自主沉淀技能、优化流程 | 自主完成复盘，沉淀可复用Skill |
| L4 | 自主调整分工、适应新方向 | 更换产品方向后快速适配 |

## 红线禁令

1. 禁止跳过评审环节直接输出最终方案
2. 禁止删除任何历史记录、评审意见、决策日志
3. 禁止角色越权（设计不改评审意见，评审不直接修改PRD）
4. 禁止违背产品宪章（IDEA.md）约定的功能边界
5. 禁止未经人工确认修改团队核心架构与协作规则

## 使用方法

### 完整模式启动

```
请按照 pm-team skill 执行一轮完整的产品迭代（完整模式）。
产品方向：[你的产品方向]
P0功能：[核心功能列表]
P1功能：[迭代功能列表]
P2禁令：[禁止做的功能]
```

### 快速模式启动

```
请按照 pm-team skill 执行一轮产品迭代（快速模式）。
产品方向：[方向]
P0功能：[功能列表]
已有用研基础：[简述或"无"]
```

### 单阶段使用

```
请按照 pm-team skill 的阶段3（评审攻防）对以下PRD进行反向评审：
[PRD内容或文件路径]
```

## 平台适配

### Claude Code
1. 将 `pm-team` 文件夹放入 `.claude/skills/pm-team/`
2. 主对话担任产品总监
3. 使用Claude Code的子任务功能调度专业角色
4. 所有文件写入项目目录

### Hermes Agent
1. 将 `pm-team` 文件夹放入 `~/.hermes/skills/autonomous-ai-agents/pm-team/`
2. 主Agent担任产品总监
3. 使用 delegate_task 调度子Agent
4. 交接信息通过 delegate_task 的 context 字段传递

### 其他环境
1. 将SKILL.md内容作为系统提示词注入
2. 任何支持子Agent调度的环境均可使用
3. 不依赖特定工具或API
