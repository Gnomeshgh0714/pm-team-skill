# 可沉淀技能模板

以下3个技能模板基于实际迭代经验提炼，可在后续迭代中直接使用。

---

## 技能1：PRD反向评审技能

### 适用场景
产品评审专员对PRD进行反向评审，发现逻辑漏洞、边界遗漏、数据一致性问题、安全风险。

### 评审5维度检查清单

| 维度 | 检查项 | 方法 |
|------|--------|------|
| 逻辑 | 状态机完整性 | 遍历所有状态，确认每个有出口，无死锁/不可达 |
| 逻辑 | 规则引擎一致性 | 检查多条规则是否重叠、优先级是否定义、边界场景是否覆盖 |
| 边界 | P0边界合规 | 搜索PRD全文，检查是否有超出边界声明的功能实现 |
| 场景 | 并发数据一致性 | 构建并发场景（如多人同时提交），验证是否有竞态条件 |
| 场景 | 极端值处理 | 检查0、1、最大值等边界输入的处理逻辑 |
| 风险 | 匿名化/双盲安全 | 检查匿名化方案是否覆盖语义级身份线索 |
| 风险 | 权限模型完整性 | 检查所有操作是否定义了角色权限 |
| 体验 | 流程闭环 | 从每个角色视角走完全流程，检查是否有断点 |

### 场景化问题分析方法
1. 阅读PRD功能描述
2. 构建极端场景（如"3位审稿人中A接受后超时，B/C拒绝"）
3. 验证PRD定义是否覆盖该场景
4. 如未覆盖或处理逻辑有矛盾，记录为问题

### 结构化输出格式
```
| 编号 | 级别 | 类别 | PRD位置 | 问题描述 |
|------|------|------|---------|---------|
| Q1 | 严重 | 逻辑漏洞 | §2.3.3 | REVIEWING->PENDING转换路径缺失... |
```

### 复审4项专项检查
1. 状态机完整性：修改后是否有新的死锁/不可达路径
2. 数据模型一致性：新增实体后所有外键引用是否正确
3. 规则引擎一致性：优先级定义后所有场景是否有明确匹配
4. 边界声明一致性：功能移除后PRD中是否还有残留引用

## 技能2：PRD修改追踪技能

### 适用场景
设计专员收到评审意见和终审修改指令后，修改PRD并保留修改痕迹。

### 修改指令结构化解析
将终审决策解析为清单：
```
| 编号 | 对应问题 | 修改要求 | 完成标准 | 优先级 |
|------|---------|---------|---------|--------|
| C-01 | Q1状态机死锁 | 补全REVIEWING→PENDING路径 | 状态流转表覆盖所有异常出口 | 必须 |
```

### 修改痕迹保留规范
- 修改内容使用标注块：`> **[V1.1修改]** 修改说明`
- 删除内容使用删除线：`~~原文~~`
- 新增内容直接写入，用标注说明
- 不删除原文，保留历史可追溯

### 修改记录文档化
在PRD文档头部增加修改记录表：
```
| 版本 | 日期 | 修改人 | 修改内容摘要 |
|------|------|--------|-------------|
| V1.0 | 2026-07-16 | 设计专员 | 初版发布 |
| V1.1 | 2026-07-16 | 设计专员 | 根据评审意见执行14项修改... |
```

### 修改完整性自检清单
1. 每项修改指令是否都有对应的修改内容？
2. 修改处是否标注了[Vx.x修改]？
3. 修改后是否有新的逻辑矛盾？
4. 版本号是否升级？
5. 旧版本是否保留？

## 技能3：总监终审裁决技能

### 适用场景
产品总监对评审意见和PRD做最终裁决。

### 逐项裁决方法
对每条评审问题独立裁决：
```
| 编号 | 问题 | 裁决 | 理由 |
|------|------|------|------|
| C-01 | 状态机死锁 | 采纳评审意见 | 阻断性逻辑漏洞，必须修复 |
| C-02 | 跨模板对齐 | 采纳方案1 | 跨模板在P0无需求场景，强制统一最简 |
```

### 修改指令分类标准
**必须修改（阻断性）**判定条件：
- 不修改会导致技术设计无法落地
- 不修改会导致上线后安全/稳定性事故
- 不修改会导致核心流程不可用

**建议修改（非阻断）**判定条件：
- 提升方案质量但不阻断流程
- 优化用户体验但不影响核心功能
- 可以在下一阶段处理的技术细节

### 关键决策记录格式
```markdown
## 决策 #XXX：[决策标题]
- 决策日期：YYYY-MM-DD
- 决策人：产品总监
- 决策类型：方案裁决

### 决策背景
[为什么需要做这个决策]

### 决策内容
[具体决策了什么]

### 决策理由
[为什么这么决策]

### 生效时间
YYYY-MM-DD，即日生效
```

### 二次终审检查清单
1. 必须修改项是否全部落实？（X/N）
2. 建议修改项落实情况？
3. 修改后是否引入新的逻辑矛盾？
4. 状态机/数据模型/规则引擎/边界声明4项专项检查
5. L1验收标准达成情况
6. 轻微非阻断问题的处理时机
