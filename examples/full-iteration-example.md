# 完整迭代示例：输入→产出对照

> 本文档展示一次完整的快速模式迭代，帮助首次使用者理解每个阶段的输入和产出。

---

## 场景设定

```
产品方向：学术文献笔记管理工具
P0功能：文献导入、笔记编辑、标签分类、全文检索
P2禁令：不做PDF阅读器、不做社交功能、不做付费会员
模式：快速模式
```

---

## 阶段1：需求梳理（快速模式，主Agent直接执行）

### 产出文件：research/20260717_需求清单.md

```markdown
# 需求清单：学术文献笔记管理工具

## 用户画像（简版）
- 目标用户：硕博研究生、高校教师
- 核心场景：阅读论文时记录笔记，按主题分类管理，后续快速检索

## 核心痛点
1. 文献散落在各处（电脑文件夹、邮箱、浏览器书签），找不到
2. 笔记和文献脱节，记了笔记但忘了对应哪篇论文
3. 标签体系混乱，不同时期用不同标签，无法统一检索
4. 检索只能按文件名，无法按笔记内容检索

## P0功能优先级
| 优先级 | 功能 | 对应痛点 | 实现难度 |
|--------|------|---------|---------|
| 1 | 文献导入（PDF/BibTeX） | P1文献散落 | 低 |
| 2 | 笔记编辑（富文本+关联文献） | P2笔记脱节 | 中 |
| 3 | 标签分类（层级标签） | P3标签混乱 | 中 |
| 4 | 全文检索（笔记+元数据） | P4检索受限 | 高 |
```

## 阶段2：PRD设计（子Agent执行）

### 产出文件：product/prd/V0.1_文献笔记管理_PRD.md

```markdown
# PRD：学术文献笔记管理工具 V0.1

## 一、需求背景
（引用需求清单中的痛点和优先级）

## 二、功能1：文献导入
### 功能描述
对应用研痛点P1。支持PDF文件导入和BibTeX格式导入...

### 用户故事
- 作为研究生，我希望拖拽PDF文件即可导入，系统自动提取标题/作者/年份...

### 功能拆解
1. PDF拖拽上传（≤50MB，格式校验）
2. 元数据自动提取（用PyPDF2读取PDF元数据）
3. BibTeX导入（.bib文件解析）
4. 文献去重（DOI/标题匹配）

### 输入输出
- 输入：PDF文件 or .bib文件
- 输出：Literature实体（literature_id, title, authors, year, doi, file_path, imported_at）

### 边界条件
- PDF无元数据：title=文件名，authors=空，提示用户手动补充
- 重复导入：DOI匹配则提示"已存在"，不重复创建
- BibTeX格式错误：逐条解析，错误条目标记并跳过

### 验收标准
- AC-1：拖拽PDF文件→3秒内完成导入→显示提取的元数据
- AC-2：导入.bib文件→解析全部条目→错误条目标记
- AC-3：重复导入同一PDF→提示已存在→不创建重复记录

## 三、数据结构设计
### Literature实体
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| literature_id | VARCHAR(64) | PK | LIT-YYYYMMDD-NNNN |
| title | VARCHAR(500) | NOT NULL | 文献标题 |
| authors | JSON | | 作者列表 |
| year | INTEGER | | 发表年份 |
| doi | VARCHAR(200) | UNIQUE | DOI（用于去重） |
| file_path | VARCHAR(500) | | PDF文件路径 |
| imported_at | TIMESTAMP | | 导入时间 |

### Note实体
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| note_id | VARCHAR(64) | PK | 笔记ID |
| literature_id | VARCHAR(64) | FK→Literature | 关联文献 |
| content | TEXT | | 富文本内容 |
| tags | JSON | | 标签列表 |
| created_at | TIMESTAMP | | 创建时间 |
| updated_at | TIMESTAMP | | 更新时间 |

## 四、提交前自检
1. ✅ 数据模型覆盖所有引用实体（Literature, Note, Tag）
2. ✅ 无状态机需求（CRUD应用）
3. ✅ 外键指向已定义实体（Note.literature_id→Literature）
4. ✅ 边界声明与实现一致
5. ✅ 无并发场景（单用户操作）
```

## 阶段3：评审攻防（子Agent执行）

### 产出文件：review/V0.1_PRD评审意见.md

```markdown
# PRD V0.1 评审意见

## 总体评价
PRD结构完整，功能拆解清晰，但存在以下问题需修改。

## 问题清单

| 编号 | 级别 | 类别 | PRD位置 | 问题描述 |
|------|------|------|---------|---------|
| Q1 | 严重 | 边界遗漏 | §二.功能1.边界条件 | PDF文件≤50MB但未定义超大文件的处理（如100MB的扫描版论文），应定义拒绝阈值和提示 |
| Q2 | 严重 | 逻辑漏洞 | §三.Literature实体 | doi字段设为UNIQUE但PDF导入时DOI可能为空，多个空DOI会违反UNIQUE约束 |
| Q3 | 一般 | 场景遗漏 | §二.功能1 | 未定义批量导入场景（用户一次拖入20个PDF），应支持批量上传 |
| Q4 | 一般 | 体验风险 | §二.功能2.用户故事 | 笔记编辑未定义自动保存机制，用户编辑过程中浏览器崩溃会丢失内容 |

## 结论：需修改
```

## 阶段4：总监终审（主Agent执行）

### 产出文件：review/V0.1_终审决策.md

```markdown
# 总监终审决策

## 逐项裁决
| 编号 | 问题 | 裁决 | 理由 |
|------|------|------|------|
| Q1 | PDF大小限制 | 采纳 | 50MB阈值合理但需定义超限提示 |
| Q2 | DOI空值UNIQUE冲突 | 采纳 | PostgreSQL中NULL不违反UNIQUE，但空字符串会，需明确空DOI存NULL |
| Q3 | 批量导入 | 采纳 | 真实场景常见，应支持 |
| Q4 | 自动保存 | 采纳 | 用户体验关键，应实现60秒自动保存 |

## 修改指令
| 编号 | 修改要求 | 完成标准 | 优先级 |
|------|---------|---------|--------|
| C-01 | 定义PDF大小超限提示 | 边界条件增加">50MB拒绝并提示" | 必须 |
| C-02 | DOI空值存NULL非空字符串 | 数据结构注释明确+边界条件说明 | 必须 |
| C-03 | 增加批量导入功能 | 功能拆解新增"批量PDF上传"子功能 | 必须 |
| C-04 | 增加自动保存 | 功能拆解新增"60秒自动保存"子功能 | 建议 |

## 结论：有条件通过，修改后定稿
```

### 修改后：定稿

设计专员按修改指令修改PRD → 升级V0.2 → 评审确认4项全落实 → 总监二次终审通过 → **定稿**。
